SYSTEM OVERVIEW - Functions and Flow (High-Level)
================================================

Scope
-----
This document summarizes how the system is structured and how a request flows through the backend, including the major functions and responsibilities of key parts. It also notes how the static frontend is served.

Top-Level Layout
----------------
- Backend/: TypeScript Express API with modular feature routers, middleware, and database access
- Frontend/: Static HTML, CSS, and JS served by the backend under /Frontend/*
- docs/, documentation/: Reference and process documents
- sql/: SQL setup, migrations, and maintenance scripts

Backend Boot Sequence
---------------------
Entry point: Backend/src/server.ts
1) Configuration: `loadEnv()` validates and loads env vars (Backend/src/config).
2) Tracing: `initTracing()` initializes OpenTelemetry tracing if enabled.
3) Database: `assertDatabaseConnection()` ensures DB connectivity; process exits on failure.
4) Bootstrap: `seedDefaultAdmin()` ensures a default admin exists (idempotent).
5) App creation: `createApp()` (Backend/src/app.ts) constructs the Express app and registers middleware/routes.
6) Server start: app.listen on HOST/PORT from env; logs "API server listening".
7) Graceful shutdown: `registerGracefulShutdown()` attaches signal handlers to close HTTP server and DB pool.

Runner Script
-------------
File: Backend/server.js
- Intended start command from Backend/: `node server.js`
- Ensures build exists (dist/server.js). If not (or FORCE_BUILD=true), runs `npm run build`.
- Delegates to compiled dist/server.js.

Express App Composition
-----------------------
Defined in: Backend/src/app.ts
- Security & Ops
  - Disable `x-powered-by`
  - Enforce HTTPS (if behind proxy and in production)
  - CORS (least-privilege) via `createCorsMiddleware()`
  - Security headers & CSP via `securityHeaders()`
  - Optional metrics via `metricsMiddleware` and /metrics endpoint (prometheus-style scrape)
  - Optional profiler (non-production) via `profilerRouter`
  - Request logging via `requestLogger`
- Body Parsers
  - `express.json({ limit: '1mb' })`
  - `express.urlencoded({ extended: true })`
- Health Endpoints (no auth, no rate limit)
  - GET /api/health → { status: 'ok' }
  - GET /api/ready → checks DB; returns ready/unready
- Static Frontend
  - Serves `/Frontend/HTML`, `/Frontend/CSS`, `/Frontend/jS` if present
- Rate Limiting
  - If not in "DEV_EASY", applies limiter under `/api`
- API Routes
  - Mounts `apiRouter` under `/api`
- Terminal Middleware
  - `notFoundHandler` for unmatched routes
  - `errorHandler` to format and log errors

Routing Structure
-----------------
Defined in: Backend/src/routes/index.ts (exported as `apiRouter`)
- Health: GET /api/health
- Auth & Admin: `/api/auth`, `/api/admin`
- Request-Type Modules (registered before general catch-all)
  - `/api/assignment`
  - `/api/assignment-termination`
  - `/api/certificate`
  - `/api/experience-certificate`
  - `/api/internal-transfer`
  - `/api/exit`
  - `/api/leave-request`
  - `/api/multi-approval` and alias `/api/approvals`
  - `/api/commissioner/tickets`
  - `/api/maternity-leave`
  - `/api/housing-allowance`
- Other Modules
  - `/api/clearance`
  - `/api/delegation`
  - `/api/onboarding`
  - `/api/roles`
  - `/api/users`
  - `/api/profile`
  - `/api/upload`
  - `/api/employee` (credentials)
- Catch-all Employee Requests
  - `employeeRequestsRouter` mounted at `/api/` last to avoid shadowing specific modules

Core and Middleware Responsibilities
------------------------------------
- Backend/src/core/*
  - database: DB pool and helpers; `assertDatabaseConnection()`
  - logger: structured logging
  - bootstrap: initial data seeding (e.g., admin user)
  - graceful: signal handling and orderly shutdown
  - middleware (under core or middleware dir): error handling, not-found, request logging
- Backend/src/middleware/*
  - CORS, security headers, HTTPS enforcement, metrics, profiler

Typical Request Flow
--------------------
1) Client sends an HTTP request to `/api/...`.
2) Global middleware runs:
   - HTTPS enforcement (if applicable)
   - CORS checks and headers
   - Security headers (incl. CSP)
   - Optional metrics collection
   - Optional profiler (non-prod)
   - Body parsing (JSON/urlencoded)
   - Request logging
3) Rate limiting (if not in DEV_EASY) is applied for `/api`.
4) Router dispatch: `apiRouter` routes the request to feature-specific routers based on the path.
5) Module handler/controller executes business logic:
   - Validates request payloads
   - Checks authentication and authorization where required
   - Interacts with the database (via core/database and module services)
   - Returns a structured JSON response or an error
6) If no route matched, `notFoundHandler` returns a 404 JSON.
7) Errors thrown by handlers are processed by `errorHandler`, which logs and returns standardized error payloads.

Frontend Flow
-------------
- The server statically serves assets:
  - `/Frontend/HTML` (no index auto-selection)
  - `/Frontend/CSS`
  - `/Frontend/jS`
- The HTML/JS pages consume the `/api/...` endpoints defined above.

Operations & Observability
--------------------------
- Health: `/api/health` (liveness), `/api/ready` (readiness with DB check)
- Metrics: `/metrics` (if enabled)
- Logs: Structured logs via core logger
- Graceful Shutdown: Ensures HTTP and DB close cleanly on process signals

Where to Extend
---------------
- Add new API surface:
  1) Create a new module under `Backend/src/modules/<feature>/`
  2) Expose a `*.routes.ts` with an Express router
  3) Register it in `Backend/src/routes/index.ts` BEFORE the catch-all `employeeRequestsRouter`
- Add middleware:
  - Place in `Backend/src/middleware/` (or `core/middleware` if cross-cutting)
  - Mount inside `createApp()` in the correct order (security/CORS before routes, error handlers last)

Notes
-----
- Order of middleware and routes is intentional for security and correctness.
- Readiness depends on DB connectivity; failures will set `/api/ready` to `503`.

Environment Variables (from Backend/src/config/env.ts)
------------------------------------------------------
Core runtime
- NODE_ENV (development|staging|production|test, default: development)
- PORT (number, default: 3037)
- HOST (string, optional)
- DEV_EASY (boolean, relaxed local dev mode; default: false)

Database
- DB_HOST (required)
- DB_PORT (number, default: 3306)
- DB_NAME (required)
- DB_USER (required)
- DB_PASSWORD (optional, default: empty)
- DB_CONNECTION_LIMIT (number, default: 10)

Auth & tokens
- JWT_SECRET (min 32 chars, required)
- ACCESS_TTL (e.g., 15m, default: 15m)
- REFRESH_TTL (e.g., 7d, default: 7d)

CORS
- CORS_ALLOWED_ORIGINS (CSV of origins; may merge from legacy CORS_ORIGINS)
- CORS_ORIGINS (legacy; merged into CORS_ALLOWED_ORIGINS if set)

Feature flags
- COMMISSIONER_SERVER_ENABLED (boolean)
- UPLOAD_SCAN_ENABLED (boolean)
- TRUST_PROXY (boolean)

Logging & rate limiting
- LOG_LEVEL (fatal|error|warn|info|debug|trace, default: info)
- RATE_LIMIT_WINDOW_MS (number, default: 900000)
- RATE_LIMIT_MAX (number, default: 100)
- ENABLE_REQUEST_LOGGING (boolean, default: false)

Bootstrap defaults (optional)
- DEFAULT_ADMIN_EMAIL
- DEFAULT_ADMIN_PASSWORD (min 8 chars)
- DEFAULT_ADMIN_NAME
- DEFAULT_ADMIN_EMPLOYEE_NUMBER
- DEFAULT_ADMIN_DEPARTMENT_CODE
- DEFAULT_ADMIN_PHONE
- DEFAULT_ADMIN_FULL_NAME_AR
- DEFAULT_ADMIN_POSITION

Uploads
- UPLOAD_MAX_SIZE (bytes, default: 10485760)
- UPLOAD_ALLOWED_TYPES (CSV of MIME types)
- UPLOAD_STORAGE_PATH (filesystem path)

Typical Requests/Responses and Error Codes
------------------------------------------
Conventions:
- Success: 200/201 with JSON payload { data, meta? }
- Validation errors: 400 { error: 'VALIDATION_ERROR', details: [...] }
- Auth: 401 { error: 'UNAUTHORIZED' } when token missing/invalid
- RBAC: 403 { error: 'FORBIDDEN' } when role insufficient
- Not found: 404 { error: 'NOT_FOUND' }
- Rate limit: 429 { error: 'TOO_MANY_REQUESTS', Retry-After: seconds }
- Server: 500 { error: 'INTERNAL_ERROR' }

Examples (abbreviated):
- POST /api/auth/login
  Request: { email, password }
  Response (200): { accessToken, refreshToken, user: { id, name, roles } }
  Errors: 400 (invalid), 401 (bad credentials), 429 (rate-limited)

- POST /api/leave-request
  Request: { type, startDate, endDate, reason }
  Response (201): { id, status: 'PENDING', ... }
  Errors: 400 (schema), 401 (auth), 403 (role), 500

- PUT /api/leave-request/:id/approve
  Response (200): { id, status: 'APPROVED' }
  Errors: 401, 403, 404, 409 (invalid state)

Authentication and RBAC
-----------------------
Auth flow:
- Login: POST /api/auth/login → returns access (short TTL) and refresh (long TTL) tokens
- Me: GET /api/auth/me with Authorization: Bearer <access>
- Refresh: POST /api/auth/refresh to rotate tokens (rate-limited)
- Revoke: POST /api/auth/revoke to invalidate refresh token (rate-limited)
- Backend expects Authorization: Bearer <access> on protected routes

RBAC:
- `authenticate` middleware validates access tokens and sets `req.auth`
- `requireRoles([...])` checks roles from token or DB (with legacy admin email fallback)
- Common roles: ADMIN, HR, MANAGER, EMPLOYEE
- Examples:
  - Admin-only: /api/admin/**, multi-approval admin endpoints, various status updates
  - Employee-only: personal request creation and “mine” listings

Role → Access Matrix (high-level)
- ADMIN: All admin endpoints, approvals, listings across modules
- HR: Most approval/status endpoints for HR workflows (leave, credentials, etc.)
- MANAGER: Approvals for team-related requests (leave, assignment)
- EMPLOYEE: Create/view own requests, profile, credentials

Data Model and Workflows
------------------------
Key entities (indicative):
- users, roles, user_roles
- requests per type (assignment, assignment_termination, internal_transfer, leave, clearance, certificate, experience, exit, maternity_leave, housing_allowance, onboarding, delegation)
- approvals and approval timelines (multi-approval)
- credentials (certificates, licenses)

Relationships (ER-style summary):
- user 1..* user_roles ..* roles
- user 1..* requests (by type)
- request 1..* approvals (multi-approval)
- user 1..* credentials (certificates/licenses)

State machines (typical):
- Request types: PENDING → APPROVED | REJECTED
- Some allow: PENDING → IN_REVIEW → APPROVED/REJECTED
- Status updates gated by roles (ADMIN/HR/MANAGER) and validated per module

Security and Limits
-------------------
CORS policy:
- Allowed origins from env `CORS_ALLOWED_ORIGINS` (CSV). In DEV_EASY, all origins allowed.
- Credentials disabled by default.

CSP/security headers:
- Helmet CSP with defaults; allows inline scripts/styles for legacy pages.
- Connect-src: 'self' plus allowed origins (and local dev hosts).
- Other headers: X-Content-Type-Options, X-Frame-Options, Referrer-Policy, Permissions-Policy.
- Report-only CSP mirrored to `/api/csp/report`.

Rate limits:
- Global `/api` limiter unless DEV_EASY is true.
- Sensitive endpoints have per-route limits (auth/login/refresh/revoke, admin user ops, approvals).

Upload limits:
- UPLOAD_MAX_SIZE default 10 MB; allowed MIME types from `UPLOAD_ALLOWED_TYPES`.
- Upload scanning optional flag `UPLOAD_SCAN_ENABLED`.
